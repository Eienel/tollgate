    How the gate works

    1. 01

      Ask

      A buyer agent requests a protected resource. The server answers HTTP 402 with payment requirements: amount, token, recipient, network.

    2. 02

      Pay

      The buyer pays in USDC on Pharos Atlantic and retries the request with an X-PAYMENT header carrying the settlement tx hash.

    3. 03

      Stamp

      Tollgate verifies the transfer on-chain, settles it idempotently, and prints a signed PAID ticket. A replay of the same tx is blocked and stamped VOID, pointing at the original.

    Why this exists

    The official Pharos x402 example gets you a demo. Shipping a real paid service runs into four gaps that Pharos documents but does not solve for you. Tollgate closes all four.

    No hosted facilitator

    Pharos has no hosted verify and settle service; the public facilitator is Base-only. Tollgate bundles a facilitator preconfigured for Atlantic with retry, idempotent settle, and a health probe.

    Idempotency left to you

    The docs require that the same on-chain tx is never billed or granted twice, and offer no turnkey solution. Tollgate makes it first class: a persistent dedupe store keyed by tx hash that survives restart.

    Sessions left to you

    The docs recommend a short-lived token after payment so you do not re-verify on-chain every request. Tollgate mints a signed session token tied to the receipt.

    Network and token confusion

    Two testnets, an internal-looking RPC, and a test USDC flagged as not official. Tollgate defines Atlantic (688689) once, pins the verified endpoints, and probes the token before trusting it.

    The STEP 0 finding

    Before building the payment path, Tollgate probes whether the Atlantic test USDC supports EIP-3009, which the gasless x402 exact scheme depends on. It does not.

    ProbeResult
    Token0xE0BE...4ec8, USD Coin, 6 decimals
    Proxy (EIP-1967 slots)None. Not a proxy
    DOMAIN_SEPARATOR()Reverts
    transferWithAuthorization in bytecodeAbsent
    permit (EIP-2612) in bytecodeAbsent
    transfer, transferFrom, approvePresent

    Verdict: a plain ERC-20, so the gasless exact scheme cannot settle against it. The payment path is an on-chain transfer the buyer broadcasts, confirmed idempotently by the bundled facilitator on settle. When Pharos ships an EIP-3009 token, only the settle mechanism changes. Reproduce it with npm run step0.

    Reliability model

    • Every payment is keyed by its settlement tx hash in an append-only log that survives restart. The first verify grants and prints PAID. Any repeat prints VOID and points at the original.
    • Settlement results are reused by tx hash on retry, never resubmitted.
    • Receipts are HMAC-signed and tamper-evident, anchored to tx hashes, with reconciliation that counts PAID once and VOID never.
    • Safety rails: dry-run mode, per-tx and per-session buyer spend caps, address and amount validation, retry with backoff, and respect for Atlantic limits (500 requests per 5 minutes, 64 pending txs per address).
    • The private key is read only from the environment and redacted from every log and error.

    Quickstart

    Requires Node 20 or newer. Runs as an MCP server over stdio by default, or HTTP.

    npm install
cp .env.example .env   # set TOLLGATE_SIGNING_SECRET, and a key for paying
npm run build
npm start              # MCP over stdio

# Or over HTTP
TOLLGATE_TRANSPORT=http TOLLGATE_HTTP_PORT=8402 npm start

    Run the full demo locally: ledger, seller, and a buyer that pays then replays.

    npm run dashboard                                  # this page, live
TOLLGATE_PAY_TO=0xYourAddress npm run seller       # a paid endpoint
TOLLGATE_PRIVATE_KEY=0xFundedKey npm run buyer -- --replay

    A typical agent flow is two calls: verify_payment to clear the ticket, then issue_access_token so the session does not re-verify on-chain every request.

    Phase 2: a paid data merchant agent

    Tollgate is the foundation for an Agent Arena entrant. The agent gates a resource with protect_endpoint, takes payment through verify_payment so it is never double-billed under load, hands buyers a session token, and watches its earnings with list_receipts. Because the skill is two-sided, the same agent pays other merchants with pay_for_resource within a budget. Every paid interaction pulls two agents through the skill: the buyer and the seller.
